
এই অংশটি Laravel অ্যাপ্লিকেশনের **ডিরেক্টরি স্ট্রাকচার** (Directory Structure) নিয়ে বিস্তারিত আলোচনা করেছে। Laravel একটি নির্দিষ্ট কাঠামো (structure) প্রদান করে যা ছোট থেকে বড় অ্যাপ্লিকেশনের জন্য কার্যকরী। তবে আপনি চাইলে আপনার অ্যাপ্লিকেশনটি আপনার পছন্দমতোও সাজাতে পারেন। Laravel এ কোন নির্দিষ্ট ক্লাস কোথায় থাকতে হবে, সে বিষয়ে খুব একটা বিধিনিষেধ নেই, যতক্ষণ না Composer সেই ক্লাসটি অটোলোড করতে পারে। আসুন, একে একে Laravel এর বিভিন্ন ডিরেক্টরি সম্পর্কে বিস্তারিত জানি।

### ১. **Root Directory (মূল ডিরেক্টরি)**

এটি আপনার Laravel অ্যাপ্লিকেশনের মূল ডিরেক্টরি, যেখানে আপনি বেশিরভাগ গুরুত্বপূর্ণ ফাইল এবং ডিরেক্টরি দেখতে পাবেন। এটি অ্যাপ্লিকেশনের সমস্ত কার্যক্রমের মূল কেন্দ্র।

### ২. **The App Directory (অ্যাপ ডিরেক্টরি)**

এই ডিরেক্টরি আপনার অ্যাপ্লিকেশনের মূল কোড ধারণ করে। Laravel অ্যাপ্লিকেশনের অধিকাংশ ক্লাস এখানে থাকবে। এই ডিরেক্টরিতে সাধারণত আপনার **Controllers**, **Models**, **Migrations**, **Middlewares**, **Requests** ইত্যাদি ক্লাস থাকবে।

### ৩. **The Bootstrap Directory (বুটস্ট্র্যাপ ডিরেক্টরি)**

এই ডিরেক্টরিতে একটি `app.php` ফাইল থাকে যা Laravel ফ্রেমওয়ার্ককে বুটস্ট্র্যাপ করে। এছাড়া, এখানে একটি `cache` ডিরেক্টরি থাকে, যেখানে ফ্রেমওয়ার্ক দ্বারা তৈরি করা ক্যাশ ফাইলগুলি যেমন রুট এবং সার্ভিস ক্যাশ ফাইল সংরক্ষিত হয়, যাতে পারফরম্যান্স অপটিমাইজ করা যায়।

### ৪. **The Config Directory (কনফিগ ডিরেক্টরি)**

এই ডিরেক্টরিতে অ্যাপ্লিকেশনের সমস্ত কনফিগারেশন ফাইল থাকে। এখানে আপনি আপনার অ্যাপ্লিকেশনের বিভিন্ন কনফিগারেশন ভ্যালু যেমন **database**, **mail**, **session** ইত্যাদি পরিবর্তন করতে পারেন। এটি গুরুত্বপূর্ণ, তাই এই ফাইলগুলি ভালভাবে দেখে নিন এবং আপনার প্রয়োজন অনুযায়ী কনফিগারেশন সম্পাদনা করুন।

### ৫. **The Database Directory (ডাটাবেস ডিরেক্টরি)**

এই ডিরেক্টরিতে আপনার **database migrations**, **model factories**, এবং **seeds** থাকে। আপনি যদি SQLite ডাটাবেস ব্যবহার করতে চান, তবে এই ডিরেক্টরিতে সেটি রাখতে পারেন।

### ৬. **The Public Directory (পাবলিক ডিরেক্টরি)**

এই ডিরেক্টরি Laravel অ্যাপ্লিকেশনের জন্য "public entry point" বা প্রথম পয়েন্ট হিসেবে কাজ করে। এখানে একটি `index.php` ফাইল থাকে, যা সমস্ত রিকোয়েস্টকে হ্যান্ডল করে এবং Autoloading সেট করে। এছাড়া, এখানে আপনার **assets** (যেমন **images**, **JavaScript**, **CSS**)ও থাকে।

### ৭. **The Resources Directory (রিসোর্স ডিরেক্টরি)**

এই ডিরেক্টরিতে আপনার **views** (যেমন Blade টেমপ্লেট), এবং **raw, uncompiled assets** (যেমন CSS বা JavaScript) থাকে। এই ডিরেক্টরি অ্যাপ্লিকেশনের ইউজার ইন্টারফেসের জন্য খুবই গুরুত্বপূর্ণ।

### ৮. **The Routes Directory (রুটস ডিরেক্টরি)**

এই ডিরেক্টরিতে আপনার অ্যাপ্লিকেশনের সমস্ত **routes** সংজ্ঞায়িত থাকে। ডিফল্টভাবে, এখানে তিনটি রুট ফাইল থাকে:

* **web.php**: এই ফাইলে সব রুট থাকে যেগুলি **web middleware group**-এ থাকে, যার মধ্যে **session state**, **CSRF protection**, এবং **cookie encryption** থাকে। এটি সাধারণত ব্যবহারকারী-ভিত্তিক রাউটের জন্য ব্যবহার হয়।
* **console.php**: এই ফাইলে আপনি **closure-based console commands** সংজ্ঞায়িত করেন। এটি HTTP রুটের মত নয়, বরং **console-based entry points** নির্ধারণ করে।
* **api.php**: এটি **stateless** রাউটগুলির জন্য ব্যবহৃত হয়। এই রাউটগুলো টোকেন দিয়ে প্রমাণীকৃত হতে হয় এবং সেগুলিতে **session state** থাকে না।
* **channels.php**: এটি ইভেন্ট ব্রডকাস্টিং চ্যানেলগুলির জন্য ব্যবহৃত হয়।

### ৯. **The Storage Directory (স্টোরেজ ডিরেক্টরি)**

এই ডিরেক্টরিতে **logs**, **compiled Blade templates**, **file-based sessions**, **file caches**, এবং অন্যান্য ফাইল থাকে যা ফ্রেমওয়ার্ক দ্বারা তৈরি হয়। এটি তিনটি সাব-ডিরেক্টরিতে বিভক্ত:

* **app**: আপনার অ্যাপ্লিকেশন দ্বারা তৈরি করা ফাইলগুলি এখানে থাকতে পারে।
* **framework**: ফ্রেমওয়ার্ক দ্বারা তৈরি করা ফাইল এবং ক্যাশ এখানে থাকে।
* **logs**: আপনার অ্যাপ্লিকেশনের লগ ফাইল এখানে সংরক্ষিত হয়।

এছাড়া, **storage/app/public** ডিরেক্টরিটি ব্যবহারকারীর জেনারেট করা ফাইল (যেমন প্রোফাইল অ্যাভাটার) রাখার জন্য ব্যবহৃত হয়। এটি **public/storage** লিঙ্কের মাধ্যমে পাবলিক অ্যাক্সেসযোগ্য হতে পারে, এবং আপনি `php artisan storage:link` কমান্ড ব্যবহার করে এটি তৈরি করতে পারেন।

### ১০. **The Tests Directory (টেস্টস ডিরেক্টরি)**

এই ডিরেক্টরিতে আপনার **automated tests** সংরক্ষিত থাকে। Laravel ডিফল্টভাবে **Pest** বা **PHPUnit** ইউটিলিটি ব্যবহার করে **unit tests** এবং **feature tests** প্রদান করে। আপনি `vendor/bin/pest` বা `vendor/bin/phpunit` কমান্ড দিয়ে আপনার টেস্ট রান করতে পারেন। অথবা, আপনি `php artisan test` কমান্ড ব্যবহার করেও এটি করতে পারেন।

### ১১. **The Vendor Directory (ভেন্ডর ডিরেক্টরি)**

এই ডিরেক্টরিটি আপনার **Composer dependencies** ধারণ করে। এটি Composer দ্বারা পরিচালিত প্যাকেজ এবং লাইব্রেরি সংরক্ষণ করে।

### উপসংহার:

Laravel-এর ডিরেক্টরি স্ট্রাকচার খুবই প্রাথমিকভাবে সাজানো হয়েছে, যা বড় এবং ছোট অ্যাপ্লিকেশন উভয়ের জন্যই কার্যকর। প্রতিটি ডিরেক্টরি একটি নির্দিষ্ট কার্যভার (responsibility) পালন করে, এবং Laravel আপনাকে এই কাঠামো অনুসরণ করতে পরামর্শ দেয়, তবে আপনি চাইলে এটি কাস্টমাইজও করতে পারেন।










এই অংশটি **Laravel অ্যাপ্লিকেশন ডিরেক্টরির** বিভিন্ন সাব-ডিরেক্টরি নিয়ে বিস্তারিত আলোচনা করেছে, যেগুলি অ্যাপ্লিকেশনের মূল কনফিগারেশন, লগিক এবং অন্যান্য কাজকে সংগঠিত করে। Laravel আপনাকে একটি স্ট্যান্ডার্ড ডিরেক্টরি স্ট্রাকচার দেয়, তবে আপনি এটি কাস্টমাইজ করতে পারেন। আসুন, এই ডিরেক্টরি গুলির প্রতিটির কার্যকারিতা এবং ভূমিকা বিস্তারিতভাবে আলোচনা করি:

### ১. **The App Directory (অ্যাপ ডিরেক্টরি)**

এই ডিরেক্টরিটি আপনার অ্যাপ্লিকেশনের মুল কোড ধারণ করে। এটি মূলত **App** নামে নেমস্পেস করে থাকে এবং Composer PSR-4 অটোলোডিং স্ট্যান্ডার্ড অনুসরণ করে অটোলোড হয়। এই ডিরেক্টরির মধ্যে আপনি দেখতে পাবেন:

* **Http**, **Models**, **Providers** ইত্যাদি ডিরেক্টরি।
* আপনি যদি Artisan কমান্ড ব্যবহার করেন, তবে বিভিন্ন সাব-ডিরেক্টরি এখানে তৈরি হবে। উদাহরণস্বরূপ, `app/Console` ডিরেক্টরি তৈরি হবে যদি আপনি `make:command` কমান্ড ব্যবহার করেন।

### ২. **The Broadcasting Directory (ব্রডকাস্টিং ডিরেক্টরি)**

এই ডিরেক্টরি শুধুমাত্র তখন তৈরি হয় যখন আপনি **make\:channel** কমান্ড ব্যবহার করেন। এটি ইভেন্ট ব্রডকাস্টিং চ্যানেল ক্লাস ধারণ করে। ব্রডকাস্টিং আপনাকে রিয়েল-টাইম ইভেন্টগুলি ক্লায়েন্টে ব্রডকাস্ট করতে সাহায্য করে।

### ৩. **The Console Directory (কনসোল ডিরেক্টরি)**

এই ডিরেক্টরি আপনার **কাস্টম Artisan কমান্ড** ধারণ করে। আপনি এখানে **make\:command** কমান্ড ব্যবহার করে নতুন কমান্ড তৈরি করতে পারেন। এই ডিরেক্টরি অ্যাপ্লিকেশনের CLI (Command-Line Interface) সম্পর্কিত সমস্ত কাজ এবং লজিক ধারণ করে।

### ৪. **The Events Directory (ইভেন্টস ডিরেক্টরি)**

এটি সাধারণত ডিফল্টভাবে থাকে না, তবে **event\:generate** এবং **make\:event** কমান্ড ব্যবহার করে এটি তৈরি করা যায়। ইভেন্টগুলি অন্যান্য অংশে আপনার অ্যাপ্লিকেশনকে জানাতে ব্যবহৃত হয় যে কোনো নির্দিষ্ট কাজ ঘটেছে। এর ফলে আপনি আরও ফ্লেক্সিবল এবং কমপ্লেক্স সিস্টেম তৈরি করতে পারেন।

### ৫. **The Exceptions Directory (এক্সেপশনস ডিরেক্টরি)**

এই ডিরেক্টরি অ্যাপ্লিকেশনের **কাস্টম এক্সেপশন** ধারণ করে। আপনি এখানে **make\:exception** কমান্ড ব্যবহার করে নতুন এক্সেপশন তৈরি করতে পারেন। এক্সেপশনগুলি সাধারণত ত্রুটি হ্যান্ডলিংয়ের জন্য ব্যবহৃত হয়।

### ৬. **The Http Directory (এইচটিটিপি ডিরেক্টরি)**

এই ডিরেক্টরি সব ধরনের **controllers**, **middleware**, এবং **form requests** ধারণ করে। সাধারণত আপনার অ্যাপ্লিকেশনে যেকোনো HTTP রিকোয়েস্ট হ্যান্ডলিং এর লজিক এখানে থাকে।

* **Controllers**: HTTP রিকোয়েস্ট হ্যান্ডল করার জন্য।
* **Middleware**: রিকোয়েস্ট প্রোসেসিংয়ের মধ্যবর্তী স্তর হিসাবে কাজ করে (যেমন, অথেন্টিকেশন বা লোগিং)।
* **Form Requests**: ফর্মের ইনপুট ভ্যালিডেশন এবং প্রক্রিয়াকরণ।

### ৭. **The Jobs Directory (জবস ডিরেক্টরি)**

এটি **make\:job** কমান্ড ব্যবহার করে তৈরি হয়। এখানে **queueable jobs** ধারণ করা হয়, যা আসলে কাজের লজিক ধারণ করে। জবগুলি অ্যাসিঙ্ক্রোনাসভাবে বা সিঙ্ক্রোনাসভাবে চলতে পারে।

### ৮. **The Listeners Directory (লিসেনার্স ডিরেক্টরি)**

এই ডিরেক্টরি **event\:generate** বা **make\:listener** কমান্ড ব্যবহার করে তৈরি হয়। এখানে **listeners** থাকে, যা ইভেন্ট ফায়ার হলে সেগুলোর হ্যান্ডলিং করে। উদাহরণস্বরূপ, **UserRegistered** ইভেন্টটি **SendWelcomeEmail** লিসেনার দ্বারা হ্যান্ডল হতে পারে।

### ৯. **The Mail Directory (মেইল ডিরেক্টরি)**

এই ডিরেক্টরি **make\:mail** কমান্ড ব্যবহার করে তৈরি হয়। এখানে সমস্ত **মেইল ক্লাস** থাকে, যেগুলি অ্যাপ্লিকেশন থেকে ইমেইল পাঠানোর জন্য ব্যবহৃত হয়। মেইল ক্লাসগুলি মেইল তৈরির লজিক ইনক্যাপসুলেট করে এবং সেগুলিকে `Mail::send` মেথডের মাধ্যমে পাঠানো হয়।

### ১০. **The Models Directory (মডেলস ডিরেক্টরি)**

এই ডিরেক্টরি আপনার **Eloquent model classes** ধারণ করে। Eloquent ORM (Object-Relational Mapping) ব্যবহারের মাধ্যমে আপনি ডাটাবেস টেবিলের সাথে ইন্টারঅ্যাক্ট করতে পারেন। প্রতিটি ডাটাবেস টেবিলের জন্য একটি মডেল ক্লাস থাকে যা সেখান থেকে ডাটা রিট্রিভ বা ইনসার্ট করতে সাহায্য করে।

### ১১. **The Notifications Directory (নোটিফিকেশনস ডিরেক্টরি)**

এই ডিরেক্টরি **make\:notification** কমান্ড ব্যবহার করে তৈরি হয়। এখানে **transactional notifications** থাকে, যা অ্যাপ্লিকেশনের বিভিন্ন ইভেন্টের উপর ভিত্তি করে ব্যবহারকারীদের নোটিফিকেশন পাঠানোর জন্য ব্যবহৃত হয়। Laravel বিভিন্ন ড্রাইভার (যেমন ইমেইল, Slack, SMS, ডাটাবেস) ব্যবহার করে নোটিফিকেশন পাঠানোর সুবিধা প্রদান করে।

### ১২. **The Policies Directory (পলিসিস ডিরেক্টরি)**

এই ডিরেক্টরি **make\:policy** কমান্ড দ্বারা তৈরি হয় এবং **authorization policy classes** ধারণ করে। পলিসি ক্লাসগুলি নির্ধারণ করে যে কোন ইউজার একটি নির্দিষ্ট রিসোর্সে কী করতে পারবে (যেমন, একটি ইউজারকে পোস্ট ডিলিট করার অনুমতি দেওয়া হবে কিনা)।

### ১৩. **The Providers Directory (প্রোভাইডারস ডিরেক্টরি)**

এই ডিরেক্টরি **service providers** ধারণ করে। **Service providers** হল Laravel অ্যাপ্লিকেশনের সমস্ত সার্ভিসের জন্য বুটস্ট্র্যাপ কোড। এখানে নতুন সার্ভিস রেজিস্টার করতে, ইভেন্টগুলি নিবন্ধন করতে, এবং অন্যান্য গুরুত্বপূর্ণ প্রস্তুতিমূলক কাজ করতে পারেন।

### ১৪. **The Rules Directory (রুলস ডিরেক্টরি)**

এই ডিরেক্টরি **make\:rule** কমান্ড দ্বারা তৈরি হয়। এখানে **custom validation rules** থাকে, যা জটিল ভ্যালিডেশন লজিককে একটি সিম্পল অবজেক্টে ইনক্যাপসুলেট করে। এটি Laravel-এর ভ্যালিডেশন সিস্টেমে ব্যবহার করা হয়।

### উপসংহার:

এই অংশটি Laravel অ্যাপ্লিকেশনের **App** ডিরেক্টরির ভিতরে থাকা বিভিন্ন সাব-ডিরেক্টরি এবং তাদের ভূমিকা নিয়ে বিস্তারিত আলোচনা করেছে। এই ডিরেক্টরি স্ট্রাকচারটি অ্যাপ্লিকেশনের মূল কোড, রাউটিং, ইভেন্টস, মেইল, মডেলস, এবং অন্যান্য কার্যকারিতা পরিষ্কারভাবে সংগঠিত করে, যাতে বড় বা ছোট অ্যাপ্লিকেশন পরিচালনা করা সহজ হয়। Laravel অ্যাপ্লিকেশনের এই ডিরেক্টরি স্ট্রাকচার আপনার কোডের স্কেল এবং মেইনটেনেবলিটি বৃদ্ধি করতে সাহায্য করে।
